# Developer (UI/UX) Reflection
**MindSpace — New Era University Student Mental Health & Wellness Tracker**

---

## Hardest Technical Problem

The hardest technical problem I ran into was getting the **post-OAuth authentication flow to play nicely with the admin route guard**. On paper it sounded simple — sign in with Google, restrict access to `@neu.edu.ph` emails, and let one whitelisted account through to `/admin` — but in practice it touched almost every layer of the app at once: Supabase auth, Google Cloud Console redirect URIs, my global `AppContext`, and the React Router guard component. Even after a successful Google login, the admin account kept getting bounced back to `/home` because of a race condition: `getSession()` would resolve with a valid session, but `onAuthStateChange` hadn't fired yet, so `currentUser` was still `null` when `AdminRouteGuard` ran its check and the fallback `role: "student"` triggered the redirect. On top of that, I was getting 401 errors from an unwhitelisted redirect URL in Supabase, and at one point I had two Supabase clients in memory simultaneously, which caused the dreaded *"Multiple GoTrueClient instances detected"* warning and inconsistent session state. Fixing it meant consolidating everything to a single `supabaseClient.js` singleton, properly configuring redirect URLs in both Supabase and Google Cloud Console, waiting for auth state to resolve before the guard evaluated, and catching a rules-of-hooks violation where a `return null` was sitting above `useState` declarations — a bug whose error message pointed nowhere near the actual problem. What made it hard wasn't any single issue; it was that auth failures cascade silently, and the symptom was three layers away from the cause.

## How I Used the AI

I used AI primarily as a **debugging partner and a second pair of eyes for architecture decisions**, not as a code generator I copy-pasted from blindly — my workflow was to build a feature myself, hit a wall, paste the exact error message and relevant file into the chat, and ask for a targeted fix rather than a rewrite, because letting the AI rewrite entire files unprompted made it easy to lose track of what changed and introduce regressions. It was most useful for lint and rules-of-hooks errors that were technically clear but confusing in practice, like the `no-unused-vars` warning for Lucide icon props in `.map()` callbacks (fixed by assigning `const StepIcon = step.icon` inside the callback body); for catching cascading import path errors after reorganizing files into subfolders, where `../hooks/` was quietly resolving to the wrong directory; for diagnosing a `ReportModal` that wouldn't appear because the parent layout had `overflow: hidden` clipping it regardless of `z-index`, which led to the `createPortal` solution; and for sanity-checking whether to migrate from Supabase to Firebase, where the AI confirmed the switch would mean rewriting the entire relational data model for no real gain. What I didn't use AI for was making design decisions — the community-style dashboard, the NEU color palette, the left panel layout, and the anonymous reporting flow all came from our own rationale; AI accelerated the implementation but didn't shape the product.

## What I Would Architect Differently

If I were starting MindSpace over, three things would change. First, I would commit to a **web-first responsive architecture from day one** instead of building everything inside a `MobileLayout` shell and scaling up — the current layout wraps every authenticated route in `TopBar`, `Sidebar`, and `BottomNav` via React Router's `<Outlet />` pattern and it works, but the shell was wireframed mobile-first, and the left panel feels underused on wider viewports because the underlying grid was never designed for 1440px; I'd start with a desktop grid and treat mobile as the adaptation. Second, I would move the **moderation and reporting logic into Supabase Edge Functions earlier** rather than leaving it as frontend-only simulated state — the report modal, onboarding flow, and post hide/unhide logic all behave correctly in the UI but don't persist or action anything in the database, which created a split-brain situation I'd rather have avoided by writing even a stub Edge Function on day one so the frontend always pointed at a real endpoint. Third, I would establish **`shared.css` and the full design token system before writing a single page component** — I built `HomePage`, `JournalPage`, `HabitsPage`, and `ProfilePage` before extracting CSS variables into a shared file, which meant every page redefined `var(--color-primary)` locally, and the cleanup showed up as subtle hex-code drift and hours of refactoring that a 30-minute upfront investment would have prevented. The common thread is the same across all three: decide the foundation before you build on top of it.
