| 10 | April 5, 2026 | Frontend Framework: React + Vite | **A:** React + Vite. **B:** Plain HTML/CSS/JS. **C:** Next.js. | All members | Chose A. The app has multiple distinct screens with dynamic state (post feeds, comment threads, report UI). Plain HTML/JS requires brittle manual DOM manipulation at this scale. Next.js adds SSR complexity (server components, app router) unnecessary for a student project with no SEO requirement. React's component model and Vite's fast dev tooling are the right fit. |
| 11 | April 5, 2026 | Database and Backend: Supabase (PostgreSQL) + Edge Functions | **A:** Supabase + Supabase Edge Functions. **B:** Firebase Firestore (no separate backend). **C:** Node.js + Express + Firebase. | All members | Chose A. Core data (Users → Posts → Comments → Reports) is inherently relational — best expressed in SQL with foreign keys, not NoSQL. Supabase provides PostgreSQL, built-in anonymous auth, and RLS policies out of the box. Firebase requires data duplication and manual cleanup for relational queries. Node.js + Express introduces a separate codebase and deployment pipeline not justified for this scope. |
| 12 | April 5, 2026 | Frontend Hosting and CI/CD: Vercel | **A:** Vercel. **B:** GitHub Pages. **C:** Netlify. | All members | Chose A. Vercel has zero-config GitHub integration and automatically generates preview deployments per Pull Request, making it easy for PM and QA to review features before merging. GitHub Pages requires manual redirect configuration for React client-side routing. Vercel's build speeds for React/Vite are faster out-of-the-box than Netlify. |
| 13 | April 5, 2026 | Authentication: Google Workspace SSO + DB-Level Domain Enforcement | **A:** Client-side email validation + standard Supabase signUp (rejected: bypassable via API). **B:** Custom server-side Edge Function registration endpoint (deprecated: required manual password management). **C:** Google Workspace SSO + PostgreSQL trigger enforcing @neu.edu.ph at DB layer. | All members | Chose C. Pivoted from Option B after resolving Docker networking issues — Option B still required students to manage passwords instead of using existing institutional SSO. Google OAuth with `hd: 'neu.edu.ph'` improves UX (no separate password). A PostgreSQL `BEFORE INSERT` trigger on `auth.users` provides zero-trust domain enforcement — even if a malicious actor strips the frontend `hd` parameter, the DB physically rejects any non-@neu.edu.ph account. |
| 14 | April 10, 2026 | Sprint 1 Frontend Development Strategy: Mock Data with Strict Schema Contract | **A:** React `useState` + `mockData.js` file using exact DB column names. **B:** Wait for backend to finish before starting frontend. **C:** Mock REST API via JSON Server (separate local server). | Dev, PM | Chose A. Waiting would have completely blocked Dev 1 and UX/UI, risking Sprint 1 deadlines. JSON Server adds setup overhead unnecessary for the UI/UX designer. Mock data using exact column names (`author_id`, `is_anonymous`, `created_at`) from Dev 2's schema ensures seamless integration in Sprint 2 — React components are already mapped to the correct data structures. |
| 15 | April 15, 2026 | Privacy Notice on Sign-Up + Institutional Email Requirement | **A:** Privacy notice shown on account creation; registration requires @neu.edu.ph email enforced at DB level. **B:** Privacy notice shown after first login (deferred). **C:** Open registration with no domain restriction. | Dev, PM, KM Analyst | Chose A. Privacy notice at sign-up is a legal and ethical baseline for a mental health application. Institutional email restriction (enforced via PostgreSQL trigger) ensures the platform remains a closed, safe community of verified NEU students only. Open registration introduces safety and moderation risks incompatible with the app's purpose. |
| 16 | April 15, 2026 | Anonymous Posting with DB-Level Security Rules | **A:** Students may post anonymously; peer users cannot see poster identity; admins can — enforced via Supabase RLS and DB views. **B:** Anonymous posts visible to no one including admins. **C:** No anonymity feature; all posts attributed. | Dev, KM Analyst, PM | Chose A. Anonymity is critical to psychological safety — students are more likely to share honestly without attribution. Full admin visibility maintains accountability. A separate DB view (`anonymous_posts_admin_view`) serves admin queries while standard RLS hides `author_id` from public-facing queries. |
| 17 | April 15, 2026 | User Profile with Self-Management Capabilities | **A:** Personal profile where students manage, update, or remove their own posts, comments, and group memberships. **B:** View-only profile (no self-management from profile page). **C:** No persistent profile; all data managed from individual content pages. | Dev, UX/UI Designer | Chose A. A centralized profile respects student privacy boundaries and provides a single place to exercise data control. Self-management maps to SECI Internalization — students actively curate their own knowledge footprint. RLS policies ensure users can only modify their own records. |
| 18 | April 27, 2026 | Community Groups by Program and Interest | **A:** Students can join or create communities grouped by program (BSCS, BSN) or shared interest (Study Tips, Basketball). **B:** Single global community feed only. **C:** Auto-assign students to groups based on program from their profile. | KM Analyst, PM, UX/UI Designer | Chose A. | 19 | April 28, 2026 | Content Moderation with Separate Admin View | **A:** Admins have a dedicated view revealing anonymous poster identities and moderation controls. **B:** Admins share the same view as regular users with no identity reveal. **C:** Automated moderation only (no admin panel). | Dev, PM | Chose A. Admin accountability access is implemented via a DB-level view that exposes `author_id` only to `service_role` queries. This preserves student trust (no peer exposure) while enabling platform integrity enforcement. Automated-only moderation lacks the judgment needed for nuanced mental health content. |
| 20 | May 1, 2026 | In-App Notification System | **A:** In-app notifications via bell icon for replies, community updates, and wellness reminders. **B:** Email notifications only (no in-app). **C:** No notification system in current scope. | Dev, PM | Chose A. | 21 | May 2, 2026 | Community Creation Gating: Multi-Gate Safety Pipeline | **Gate 1 — Account Standing:** Block creation if account is under 7–30 days old, profile is incomplete, or user has prior moderation strikes. **Gate 2 — Content Screening:** Run group name, description, and topic tags through an NLP filter before submission. **Gate 3 — Rate Limiting:** Cap each user at 1–2 group creations per month. **Gate 4 — Human Moderator Review:** No group goes live automatically — all submissions enter a moderator review queue. **Gate 5 — Probation Period:** New groups start with a member cap (10–20 people) and elevated monitoring for 2–4 weeks. | Dev, PM, KM Analyst | Chose Gate 4 (Human Moderator Review) as the primary enforced gate, supported by Gates 1, 3, and 5 as automated layers. Gate 4 is non-negotiable on a sensitive mental health platform — automated filters alone cannot evaluate intent or context. Gates 1 and 3 reduce spam volume reaching the review queue. Gate 5 limits blast radius if a bad-faith group slips through initial review. Gate 2 (NLP content screening) is noted as a future enhancement; a keyword blocklist is feasible in scope but the full Moderation API integration is deferred. |
