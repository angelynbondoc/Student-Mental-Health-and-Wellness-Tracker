**Entry 3: Fixing State Persistence Bugs (Lifting State Up)**

* **What I learned or decided:** I gained a deep, practical understanding of the React component lifecycle and why local state dies when a component unmounts. I also learned that while passing props down (prop drilling) works for Sprint 1 mock data, managing complex relational data (like tying comments to a specific post ID) via top-level arrays is extremely messy. This reinforced why we need Dev 2's Supabase database for Sprint 2.

* **Date & Task:** April 9, 2026 - Building the refactored Batch 1 features (Mobile Layout, Community Feed, Create Form, and Reactions/Comments) using top-level global state to match the final Supabase schema.
* **The Prompt:** "Task - Build Batch 1 (Core Social & App Shell)... CRITICAL SETUP: In `App.jsx`, create a mock `currentUser` object... Initialize top-level `useState` arrays for communities, profiles, posts, comments, and reactions. Feature 1: App Navigation. * **What the AI produced:** The AI successfully generated all 6 requested files (`mockData.js`, `App.jsx`, `MobileLayout.jsx`, `HomePage.jsx`, `CreatePage.jsx`, and `PostCard.jsx`). It correctly implemented the complex relational logic, such as using `.find()` to cross-reference the `profiles` array for author names and `.filter()` to count upvotes from the `reactions` array. It also heavily commented the code as requested for my oral defense prep.
* **What I changed, rejected, or improved — and WHY:** I reviewed the AI's state update logic in `CreatePage.jsx` and the Reaction toggle logic. The AI originally just passed `posts` and `setPosts`, but I updated the submit functions to use the functional state update method (e.g., `setPosts(prevPosts => [newPost, ...prevPosts])`). I made this change because using previous state callbacks guarantees we don't accidentally encounter "stale closures" (overwriting data if multiple state updates happen at the exact same time). I also prepended `newPost` so it appears at the top of the feed rather than the bottom.
* **What I learned or decided:** I learned how to simulate SQL `JOIN` operations entirely on the frontend using React array methods. By mapping over `posts` and finding matching IDs in the `profiles` and `communities` arrays, I effectively created a relational database in the browser. This deeply helped me understand how Dev 2's backend queries will eventually work in Sprint 2.

**Entry 6: Building Batch 2 (Personal KM & Habit Tracking)**
* **Date & Task:** April 9, 2026 - Expanding the global state and building Batch 2 features (Mood Journal, Resource Library, Habit Tracker, and Community Filtering) to support SECI Externalization and Combination.
* **The Prompt:** "Task - Build Batch 2 (Personal KM & Community Groups)... CRITICAL SETUP: In `App.jsx`, add new top-level `useState` arrays for: `mood_journal`, `resources`, `habits`, and `habit_logs`... Feature 5: Mood Journal... Feature 6: Resource Library... Feature 7: Habit Tracker (Add a 'Log Today' button... Prevent duplicate logging for the same day)... Feature 8: Community Groups (Add a way to filter the feed by `community_id`)..."
* **What the AI produced:** The AI updated `mockData.js` and `App.jsx` with the four new state arrays. It generated `JournalPage.jsx`, `ResourcesPage.jsx`, and `HabitsPage.jsx`. It also updated the `HomePage` feed with a `.filter()` method to show only posts matching the selected community. It heavily commented the Habit Tracker duplicate prevention logic as requested.
* **What I changed, rejected, or improved — and WHY:** I had to fix a major bug in the AI's duplicate logging logic for the Habit Tracker. The AI tried to check for duplicates by comparing `new Date().toISOString()`. However, ISO strings include the exact time down to the millisecond, meaning the check would never find a match and users could log the habit infinitely! I rewrote the check to split the timestamp and only compare the calendar date (e.g., `existingLog.completed_date.split('T')[0] === new Date().toISOString().split('T')[0]`). I also had to manually add the `<Route path="/habits">` in `App.jsx` because the AI forgot to update the router.
* **What I learned or decided:** I learned that handling dates and times in JavaScript requires extreme precision, especially when comparing timestamps. I also realized that doing array `.filter()` operations for the Community feed is perfectly fine for mock data, but doing this client-side with 10,000 real posts would severely lag the app. This solidified my understanding of why Dev 2 will need to handle data filtering on the backend via Supabase queries during our Sprint 2 integration.
